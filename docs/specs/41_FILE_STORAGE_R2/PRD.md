# 41 — File Storage with Cloudflare R2 — PRD

**Status:** 📝 Draft
**Last updated:** 2026-05-18

## Problem Statement

RevLooper needs a secure, scalable file storage system for:
- AI Brain document uploads (PDF, DOCX, TXT, MD)
- Lead CSV imports (bulk lead data)
- Campaign content assets (images, videos generated by Content Studio)
- Email attachments (future)
- User profile avatars (future)

Current state: No file storage implementation exists. Frontend components like `UploadDocModal.tsx` only handle text content, not binary files.

## User Stories

### As a workspace admin
- I want to upload documents to my AI Brain so the AI can reference my company knowledge
- I want to import leads via CSV so I can bulk-add prospects to campaigns
- I want my uploaded files to be private so competitors can't access my data
- I want file uploads to be fast (<3s for 10MB file from Vietnam/Thailand)

### As a campaign manager
- I want to upload brand assets (logos, images) so Content Studio can use them in generated content
- I want to download generated content assets so I can use them in external tools
- I want file URLs to expire so shared links don't leak data permanently

### As a developer
- I want a simple REST API for file operations so I can integrate storage into any service
- I want workspace isolation enforced at the storage layer so cross-tenant leaks are impossible
- I want file metadata queryable via SQL so I can build file management UIs

## Functional Requirements

### FR-1: File Upload Flow (Presigned POST)
1. Client calls `POST /v1/storage/upload/initiate` with:
   - `filename` (original filename with extension)
   - `content_type` (MIME type)
   - `file_size` (bytes)
   - `purpose` (enum: `ai_brain_doc`, `lead_import`, `content_asset`, `avatar`)
2. Storage-service validates:
   - File size ≤ 100MB
   - Content type is in whitelist (see FR-4)
   - Workspace has storage quota remaining
3. Storage-service generates:
   - Random UUID for file ID
   - Date-based path: `{workspace_id}/{yyyy}/{mm}/{dd}/{file_id}/{filename}`
   - Presigned POST URL (15-minute expiry)
   - Presigned POST fields (policy, signature, key, etc.)
4. Storage-service writes metadata row to `storage_files` table (status=`pending`)
5. Client uploads file directly to R2 using presigned POST
6. Client calls `POST /v1/storage/upload/complete` with `file_id`
7. Storage-service updates metadata row (status=`ready`)

### FR-2: File Download Flow (Presigned GET)
1. Client calls `GET /v1/storage/files/{file_id}/download`
2. Storage-service validates:
   - File exists and belongs to requester's workspace
   - File status is `ready`
3. Storage-service generates presigned GET URL (30-minute expiry)
4. Client downloads file directly from R2 using presigned URL

### FR-3: File Metadata Management
- `GET /v1/storage/files` — list files (paginated, filterable by purpose/status)
- `GET /v1/storage/files/{file_id}` — get file metadata
- `DELETE /v1/storage/files/{file_id}` — soft-delete file (marks as `deleted`, actual R2 deletion via lifecycle policy)

### FR-4: Supported File Types (Whitelist)
| Purpose | Allowed MIME Types | Max Size |
|---|---|---|
| `ai_brain_doc` | `application/pdf`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `text/plain`, `text/markdown`, `text/csv` | 50MB |
| `lead_import` | `text/csv`, `application/vnd.ms-excel`, `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` | 10MB |
| `content_asset` | `image/png`, `image/jpeg`, `image/webp`, `application/pdf` | 100MB |
| `avatar` | `image/png`, `image/jpeg`, `image/webp` | 5MB |

### FR-5: Path Structure
```
{workspace_id}/{yyyy}/{mm}/{dd}/{file_id}/{filename}
```

Example:
```
a1b2c3d4-e5f6-7890-abcd-ef1234567890/2026/05/18/f9e8d7c6-b5a4-3210-fedc-ba9876543210/company-profile.pdf
```

**Rationale:**
- `{workspace_id}` prefix enforces workspace isolation at R2 bucket level
- Date-based structure enables lifecycle policies (auto-delete files older than N days)
- Random `{file_id}` prevents enumeration attacks (122-bit UUID entropy)
- Original `{filename}` preserved for download Content-Disposition header

### FR-6: Storage Quotas (by Plan)
| Plan | Storage Limit | Files Limit |
|---|---|---|
| Free | 100MB | 50 files |
| Starter | 1GB | 500 files |
| Pro | 10GB | 5,000 files |
| Enterprise | 100GB | 50,000 files |

Quota enforcement:
- Checked on upload initiation (before presigned URL generation)
- Soft-deleted files count toward quota until purged (7-day retention)

### FR-7: File Lifecycle
1. **Pending** — metadata created, presigned URL issued, awaiting upload
2. **Ready** — file uploaded to R2, available for download
3. **Deleted** — soft-deleted by user, hidden from listings, purged after 7 days
4. **Failed** — upload never completed (pending >24 hours), marked as failed by cleanup job

## Non-Functional Requirements

### NFR-1: Performance
- Upload initiation API response: <200ms (p95)
- Presigned URL generation: <50ms (p95)
- File upload to R2: <3s for 10MB file from SEA region (p95)
- Download URL generation: <100ms (p95)

### NFR-2: Security
- All files stored in private R2 bucket (no public access)
- Presigned URLs use HMAC-SHA256 signatures
- Upload URLs expire in 15 minutes
- Download URLs expire in 30 minutes
- Workspace isolation enforced at path prefix level
- File type validation via MIME type + magic byte verification

### NFR-3: Reliability
- Upload completion idempotent (duplicate `complete` calls no-op)
- Orphaned files (pending >24h) cleaned up by daily job
- Soft-deleted files purged after 7 days
- R2 provides 99.9% durability (11 nines)

### NFR-4: Scalability
- Support 100 workspaces × 1,000 files = 100,000 files
- Support 10 concurrent uploads per workspace
- R2 bucket lifecycle policy auto-deletes files >365 days old

## Acceptance Criteria

### AC-1: Upload Flow
- [ ] Client can initiate upload and receive presigned POST URL
- [ ] Client can upload file directly to R2 using presigned POST
- [ ] Client can complete upload and file becomes `ready`

- [ ] Upload fails if content type not in whitelist

### AC-2: Download Flow
- [ ] Client can request download URL for owned file
- [ ] Client can download file from R2 using presigned GET URL
- [ ] Download fails if file belongs to different workspace
- [ ] Download fails if file status is not `ready`
- [ ] Download URL expires after 30 minutes

### AC-3: Metadata Management
- [ ] Client can list files with pagination
- [ ] Client can filter files by purpose and status
- [ ] Client can soft-delete file
- [ ] Soft-deleted files hidden from listings
- [ ] Soft-deleted files purged after 7 days

### AC-4: Security
- [ ] Cross-tenant access blocked (workspace A cannot access workspace B's files)
- [ ] Invalid MIME types rejected
- [ ] Expired presigned URLs rejected by R2
- [ ] File path includes workspace_id prefix

### AC-5: Quota Enforcement
- [ ] Upload blocked when storage quota exceeded
- [ ] Upload blocked when file count quota exceeded
- [ ] Quota calculated correctly (excludes purged files)

## Out of Scope (Phase 1)

- Virus/malware scanning (Phase 2 — ClamAV integration)
- Image resizing/optimization (Phase 2 — handled by image-gen-service)
- CDN caching for public assets (all files private in Phase 1)
- File versioning (single version per file)
- Resumable uploads (use presigned POST, not multipart)
- File sharing between workspaces
- Public file URLs (all files require presigned URLs)

## Dependencies

- Cloudflare R2 bucket provisioned (via Terraform)
- R2 API credentials stored in GCP Secret Manager
- `workspace_id` available in request context (from api-gateway)
- Billing-service feature gate for storage quotas

## Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| R2 outage blocks all uploads | HIGH | Implement retry logic with exponential backoff; show user-friendly error |
| Presigned URL enumeration attack | MEDIUM | Use random UUIDs (122-bit entropy); monitor for suspicious access patterns |
| Storage quota abuse | MEDIUM | Enforce quotas at API layer; alert on workspaces approaching limits |
| Orphaned files (upload initiated but never completed) | LOW | Daily cleanup job marks pending >24h as failed; lifecycle policy deletes after 30 days |
| MIME type spoofing (upload .exe as .pdf) | MEDIUM | Validate magic bytes, not just extension; reject mismatches |
